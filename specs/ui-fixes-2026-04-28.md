## 1. Share code: switch to all-digits + add manual entry to Join Sale

**Problem.** Current share codes are 6 chars from `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (alphanumeric, no 0/O/1/I/L). Mixed-case letter+digit entry on iOS forces a full keyboard, which is hostile to a 65yo user. The Join Sale instruction sheet (`#join-instruction-modal`) is just a "scan a QR" tutorial with a "Got It" button — no fallback path if the camera fails or permission is denied.

**Solution.**

### 1a. Backend (`api/src/worker.js`)
- Rewrite `generateShareCode()` (line 46) to return 6 random digits `0-9`. Drop the disambiguation logic — there's no visual confusion in pure digits.
- **Keep the URL routing regex `[A-Z0-9]+` unchanged** (line 323). Pure digits are a subset, so legacy alphanumeric codes already in D1 continue to resolve.
- **No D1 migration.** Existing sales keep their alphanumeric codes; only newly created sales get digits. Both work.
- Acknowledge address-space tradeoff in a code comment: 1M codes (down from ~1B). Acceptable for current scale; revisit if collisions appear in metrics.

### 1b. Frontend — add manual entry to Join Sale instruction sheet (`#join-instruction-modal`, `index.html:594-604`)
Restructure the sheet:
- Title stays: "Join a Sale"
- Primary instruction stays: "Scan the QR code shown on the organizer's phone with your camera app."
- Helper line stays: organizer hint about Share Sale.
- **NEW:** A divider labeled "or enter the code"
- **NEW:** A 6-digit input field. `<input type="tel" inputmode="numeric" pattern="[0-9]*" maxlength="6" autocomplete="off">` — `type="tel"` reliably triggers the iOS numeric keypad without bringing up autofill clutter.
- **NEW:** Auto-submit when 6 digits are entered. No "Join" button. On submit, call the existing `Sync.fetchSaleByCode(code)` flow and feed into `confirmJoinSale`.
- Loading state: dim the input + show a small spinner inline while fetching.
- Error state: if the code doesn't resolve, show inline red text under the input ("That code isn't active. Check with the organizer.") and clear the input for retry.
- Remove the existing "Got It" button — the manual input is now the secondary path; nothing to acknowledge.
- Closing the sheet remains: tap backdrop or swipe down (existing pattern via `.overlay`).

### 1c. Backend regex sanity check
The route `/sales/by-code/:shareCode` regex `[A-Z0-9]+` is case-sensitive (no `i` flag). The frontend must uppercase user input before sending — but for digit-only input there's nothing to uppercase. Defensive: in the new manual-entry handler, still call `.toUpperCase()` on the input so legacy alphanumeric codes someone shares work too.

**Files touched:**
- `api/src/worker.js` — `generateShareCode()`
- `index.html` — `#join-instruction-modal` markup
- `js/app.js` — `showJoinInstruction`, new manual-submit handler, `confirmJoinSale` integration
- `css/styles.css` — minor input/divider styles within the join sheet

**Acceptance criteria:**
- New sale is created → share code is 6 digits.
- Old alphanumeric code in D1 → still resolves and joins correctly.
- User taps Join Sale on Setup → instruction sheet opens with a numeric input.
- User enters 6 digits → join confirmation appears (or error if invalid).
- iOS Safari shows the numeric keypad on focus.
- Sheet still scrolls cleanly; no content cut off.

## 2. Remove swipe-to-delete; replace with explicit Delete in edit flows

**Problem.** Two persistent swipe-to-delete patterns exist (`checkout.js:972 bindSwipeToDelete` for items, `sale-setup.js:243 _bindSwipeToDelete` for day rows). They:
- Visually collide with right-edge content during the swipe (e.g., red X overlays "+ Add Discount" link on day rows).
- Require a fading "Swipe left to delete" hint to be discoverable — exact pattern the design philosophy rejects ("teach by doing, not by reading").
- Are unreliable on mobile Safari near the back-swipe edge.

**Solution.**

### 2a. Item rows (Order #1 sheet)
- Remove `.item-row--swipeable` class, `.item-row__delete-bg` markup, `bindSwipeToDelete` function, click handler on `[data-swipe-delete]`, and `showSwipeHint`/`.swipe-hint` CSS (`checkout.js` lines 925-1042 area; `styles.css` `.swipe-hint`, `.item-row__delete-bg`, `.item-row__content` transform-related styles).
- Tap on item row continues to open the Add Item screen in edit mode (`openEditItemSheet`, line 500). This already works.
- **NEW:** In edit mode, add a red **"Delete Item"** button at the bottom of the Add Item screen. Hard delete style — matches `.sheet__btn--danger` color tokens. Sits above the green "Save Changes" button OR replaces the existing footer layout to show both. Recommended: stack vertically inside `.entry-screen__footer`, Save on top, Delete below.
- Hide the Delete button when the screen is in Add mode (`editingItemIndex === null`); show only when editing.
- Tap Delete → no confirm dialog (item-level delete is low-stakes; user can re-add). Just remove and close back to Order #1 sheet.

### 2b. Day rows (Setup screen)
- Remove `_bindSwipeToDelete`, `.discount-row__delete-bg` markup, click handler at line 232 (`sale-setup.js` lines 226-287, related CSS).
- Day rows already open `_openDiscountEdit` on tap of the right-side discount text (`sale-setup.js:200`). Extend the inline edit area to include a **Delete Day** affordance — but only when `scheduleDays.length > 1` (preserve the existing `canDelete` rule from line 227).
- Recommended: small "Remove this day" link in red, right-aligned below or beside the discount input. Tappable, no confirm. After tap, calls existing `_deleteDay(index)`.

### 2c. Cleanup
- Delete CSS rules for `.swipe-hint`, `.item-row--swipeable`, `.item-row__delete-bg`, `.discount-row__delete-bg`, and any transform/transition styles that only exist for swipe.
- Verify no references remain via grep before committing.

**Files touched:**
- `js/checkout.js` — remove swipe code, add Delete button in edit mode
- `js/sale-setup.js` — remove swipe code, add Delete in discount edit flow
- `index.html` — Add Item footer markup change (add Delete button shell)
- `css/styles.css` — remove swipe-related rules, add Delete button placement

**Acceptance criteria:**
- Tap an item in Order #1 → Edit Item screen opens with Save Changes + Delete Item buttons.
- Tap Delete Item → item removed, Order #1 sheet refreshes, no confirm needed.
- Tap a day row's discount → inline edit opens with Remove option (only if 2+ days).
- No swipe gesture has any effect on either row type.
- No "Swipe to delete" hint ever appears.
- Grep returns zero hits for `bindSwipeToDelete`, `swipe-hint`, `item-row__delete-bg`, `discount-row__delete-bg`.

**Problem.** Mic button (`#add-item-mic`), Qty − (`#add-item-qty-minus`), Qty + (`#add-item-qty-plus`), and numpad backspace (`.numpad__button--backspace`) have icons that aren't visually centered in their circles.

**Solution.**

- The `&minus;` and `+` glyphs in qty buttons (`index.html:402,404`) are text characters — they need `line-height: 1` and a precise `font-size` to center. Alternative: replace with SVG icons for true geometric centering.
- Recommendation: convert qty − / + to small SVG icons (matching the design system's icon language) for a uniform circle-icon look.

## 6. Dashboard — consolidate filter pills + sort onto one row

**Problem.** Five filter pills (All/Open/Unpaid/Paid/Void rendered by `dashboard.js:281 renderFilterPills`) overflow on mobile (image 5 cuts off "Paid" + Void). Sort sits on its own line (`#dashboard-sort`), wasting vertical space.

**Solution.**

### Markup
Replace the current `dashboard-controls` block (`index.html:285-293`):
```html
<div class="dashboard-controls">
  <div class="dashboard-controls__top">
    <button id="dashboard-sort">Newest First ↓</button>
  </div>
  <div class="dashboard-filters" id="dashboard-filters"></div>
</div>
```

  <button class="dashboard-pill" id="dashboard-filter-trigger">

    <svg>chevron-down</svg>
  </button>
  <button class="dashboard-pill" id="dashboard-sort">Newest First ↓</button>

### Filter sheet
Tapping `#dashboard-filter-trigger` opens a new bottom sheet (`#dashboard-filter-modal`) listing all 5 filters with their counts. Use the existing `.ec-picker-list` pattern (`styles.css:1592` "ec-picker-list (V2 §2.2 — bottom-sheet selection rows)") for visual consistency.

### Trigger label format

### Behavior
- Sort pill behavior unchanged — single tap toggles direction.
- `renderFilterPills` is replaced by a much smaller `renderFilterTrigger()` that just updates the label/count on `#dashboard-filter-label`.
- `resetFilters()` continues to set `activeFilter = 'all'`, sortNewestFirst = `true`.

**Files touched:**

## 9. Sheet spacing — global token + audit

**Problem.** Sheet title-to-content spacing is ad-hoc across the app:
- `.sheet__title` has `margin-bottom: var(--space-md)` (`styles.css:2006`) — about 12-16px depending on token value.
- `.consignor-form__label:first-of-type { margin-top: 0 }` (`styles.css:1486`) makes the first label sit flush against the title's margin-bottom — too tight (image 7, image 8).
- Sale Confirmation sheet has the same issue (image 8): "Sale Confirmation" → "Name (none)" is cramped.

**Solution.**
- Define a new design token in `:root` (`styles.css` or wherever tokens live): `--sheet-content-gap: 24px`.
- Update DESIGN_SYSTEM.md to document it.
- Apply it: `.sheet__title { margin-bottom: var(--sheet-content-gap); }` — replaces the current `var(--space-md)`.
- Audit every sheet's first child element. If any sheet has its own first-child margin override (like `.consignor-form__label:first-of-type { margin-top: 0 }`), keep that — the sheet title's margin-bottom is now the canonical gap.
- Verify in particular: Add Consignor, Sale Confirmation, Edit Sale, Haggle, Item-detail (Order #1), Speech Confirm, Discount edit, all picker sheets.

**Files touched:**
- `css/styles.css` — token definition, `.sheet__title` margin-bottom
- `DESIGN_SYSTEM.md` — document the token under the existing sheet section

**Acceptance criteria:**
- Visual breathing room above all sheet content matches across every sheet.
- No sheet has flush-against-title content.
- `--sheet-content-gap` resolves to a single value used by all sheet titles.
